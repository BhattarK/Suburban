## Recommended Approach for Oracle XML input

### Yes — it is possible

You can keep the same database tables (`Branch`, `Document`, `Document_Line`) and the same Quadient workflows, while replacing the fixed-width TXT parsing with Oracle XML parsing.

## Recommended approach

### 1. Add XML ingestion alongside the existing TXT flow
- Keep `suburban_propane.py` as the main workflow.
- Extend `get_input_filenames()` to detect `.xml` files in the input source.
- Treat XML files the same as TXT files from the perspective of downstream processing:
  - download to local folder
  - archive originals
  - skip already-processed filenames in `datafile`

### 2. Create an XML parsing adapter
- Add a new parser module, e.g. `dependencies/xml_parser.py`
- Use `xml.etree.ElementTree` or `lxml` to read the Oracle XML structure.
- Convert XML content into the same logical data model the pipeline currently expects:
  - one Branch per invoice/document group
  - one or more Document rows per branch
  - Document_Line rows for line items

### 3. Preserve the current DB ingestion path
- The existing `process_input_file()` already handles:
  - `Branch` insert
  - `Document` insert
  - `Document_Line` insert
  - PDF filename generation
  - mail manager creation
- Reuse that path by feeding it structured data from XML instead of raw TXT records.

### 4. Map XML to the same schema
- Create a mapping layer from XML element/attribute names to DB column names.
- You can reuse the current JSON schema concept, but for XML:
  - define `path`, `column`, `type`, `tableType`
  - or write custom parser logic for each file type
- Example:
  - `INV` XML: map invoice header fields to `Branch`
  - map line items to `Document_Line`
  - preserve the `document_date`, `customer_account_number`, `csc_number`, etc.

### 5. Determine file type from XML
- Identify `INV`, `STA`, `REN` from:
  - filename pattern
  - XML root element
  - a type attribute inside the XML
- Then select the right helper logic:
  - `dependencies/file_type_code/inv_code.py`
  - `dependencies/file_type_code/sta_code.py`
  - `dependencies/file_type_code/ren_code.py`

### 6. Keep Quadient and mail workflow unchanged
- Once DB rows are populated the same way:
  - `create_pdfs()` can still call `run_InspirePSCLI()`
  - mail-manager output files can still be generated by `dependencies/mail_manager.py`
- This is the main advantage: preserve the same output pipeline.

## Suggested solution structure

### Option A — Add XML support into existing pipeline
- Extend `suburban_propane.get_input_filenames()`
- Add `parse_xml_file(input_filename, file_type)`
- Feed parsed data into `process_input_file()` with a new XML branch
- Maintain TXT processing for backward compatibility

### Option B — Create a separate XML-to-DB adapter
- Implement `oracle_xml_to_db.py`
- This script:
  - reads XML
  - converts it into the same DB inserts
  - calls the same `mail_manager`, `get_file_data()`, `create_pdfs()` steps
- This is cleaner if the XML structure is very different from TXT.

## Practical implementation notes
- Use `ElementTree.parse(input_filename)` or `lxml.etree` if XML namespaces are involved.
- Normalize values before SQL:
  - dates
  - numeric fields
  - escaped text
- Keep `clean_string()` from `dependencies/utility_functions.py` for DB-safe quoting.
- Preserve `sequence_number.txt` and PDF file naming strategy.

## Potential issues to watch
- Oracle XML may have nested structures that do not map one-to-one to current record types.
- If XML includes new fields, decide whether they map into existing DB columns or require schema additions.
- Ensure the XML parser handles:
  - namespaces
  - optional/missing fields
  - repeated line items
  - encoding differences

## High-level recommendation

1. Keep the current pipeline and workflows.
2. Replace only the input parsing layer.
3. Build an XML adapter that outputs the same internal data model.
4. Reuse existing DB insert logic and Quadient workflow invocation.
5. Keep old TXT parsing until you are sure XML is fully working.
